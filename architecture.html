Tech Radar — Architecture

C4 model view of the Tech Radar system: structure diagrams at three levels of zoom, plus behaviour (sequence) diagrams for the three main workflows.

L1System Context

Who uses the radar, and what external systems does it depend on? Two personas interact with the system: an Engineer who runs and curates it, and a Team Member who views the resulting dashboard. Three external data sources feed the discovery pipeline.

C4Context title Tech Radar — System Context Person(engineer, "Engineer / Architect", "Runs discovery, curates items, builds the site") Person(viewer, "Team Member", "Browses the dashboard to track technology choices") System(radar, "Tech Radar", "Discovers emerging technologies, classifies them through a curation workflow, and serves an interactive static dashboard") System_Ext(github, "GitHub Trending", "Public trending repos page — no auth required") System_Ext(reddit, "Reddit", "Public JSON endpoints for r/programming, r/rust, r/devops, etc.") System_Ext(rss, "RSS / Atom Feeds", "Tech news and blog feeds") Rel(engineer, radar, "Runs scripts, curates via CLI") Rel(viewer, radar, "Views dashboard in browser") Rel(radar, github, "HTTP scrape, daily") Rel(radar, reddit, "Public JSON API") Rel(radar, rss, "HTTP feed fetch")

L2Containers

The system is made of five distinct runnable/deployable units and two data stores. Three Python scripts form the backend pipeline. radar_core.py is a shared library used by all three. data/ is a flat filesystem store (no database). The Web Dashboard is a fully static React app served from site/.

L3Component — Discovery Engine

Inside runner.py and the scrapers/ package. The Orchestrator drives the pipeline: run each scraper, deduplicate via a canonical-URL index built from existing items, persist new items, then assemble the dashboard manifest. Note: the RSS scraper is planned but not yet implemented.

C4Component title Discovery Engine — Components (runner.py + scrapers) Container_Boundary(runner_bound, "Discovery Runner") { Component(orch, "Orchestrator", "runner.py", "Iterates over registered scrapers, routes results through dedup, triggers the manifest rebuild") Component(gh_scraper, "GitHub Trending Scraper", "scrapers/github_trending.py", "Parses github.com/trending HTML with stdlib HTMLParser; extracts name, language, stars gained, description") Component(reddit_scraper, "Reddit Scraper", "scrapers/reddit.py", "Calls public Reddit .json endpoints for configured subreddits; extracts linked GitHub repos for cross-source dedup") Component(rss_scraper, "RSS Scraper [planned]", "scrapers/rss_feeds.py", "Will parse RSS/Atom feeds from configured tech news sources") Component(dedup, "Dedup Engine", "radar_core.py", "Builds a canonical-URL index from all stored items; detects when a new discovery is the same repo seen elsewhere") Component(persist, "Item Persister", "radar_core.py — save_new, touch_last_seen, record_also_seen", "Writes new items; refreshes last_seen and star history on re-encounters; merges also_seen for cross-source matches") Component(builder, "Manifest Builder", "runner.py — build_radar_json", "Walks data/items/, computes star trend for each item, sorts, writes data/radar.json") } ContainerDb(items, "Item Store", "JSON files") ContainerDb(manifest_db, "Radar Manifest", "JSON file") ContainerDb(raw, "Raw Audit Log", "JSON files per scraper per day") Rel(orch, gh_scraper, "discover()") Rel(orch, reddit_scraper, "discover()") Rel(orch, rss_scraper, "discover() [planned]") Rel(orch, dedup, "build canonical index; check each result") Rel(orch, persist, "save_new / touch_last_seen / record_also_seen") Rel(orch, raw, "write raw scrape output for audit") Rel(orch, builder, "call after all scrapers complete") Rel(dedup, items, "read all canonical_url fields") Rel(persist, items, "read/write per-tech JSON files") Rel(builder, items, "read all JSON files") Rel(builder, manifest_db, "write radar.json")

L3Component — Curation CLI

Inside radar.py and radar_core.py. The CLI is structured around a set of focused command handlers wired to an argparse subcommand parser. Every mutating command ends with a Radar Rebuild so the dashboard manifest stays in sync.

C4Component title Curation CLI — Components (radar.py) Container_Boundary(cli_bound, "Curation CLI") { Component(parser, "CLI Argument Parser", "argparse subcommands", "Routes list / show / find / promote / demote / set / archive to the right handler") Component(resolver, "Item Resolver", "radar.py — _resolve()", "Finds a single item by exact id, exact name, or unique partial name; exits with an error on ambiguity") Component(viewer, "Viewer", "cmd_list, cmd_show, cmd_find", "Read-only commands: list grouped by ring with momentum sorting; show full JSON; full-text search") Component(ring_mgr, "Ring Manager", "cmd_promote, cmd_demote", "Moves an item between rings; demote always sends back to Discovered") Component(field_editor, "Field Editor", "cmd_set", "Sets quadrant, company, description, or momentum; validates quadrant values against the allowed list") Component(archiver, "Archive Manager", "cmd_archive", "Sets ring to Hold and appends the 'archived' tag so re-discovery is suppressed") Component(rebuilder, "Radar Rebuilder", "_rebuild() — delegates to runner.build_radar_json", "Regenerates data/radar.json after every mutating command") } ContainerDb(items_c, "Item Store", "JSON files") ContainerDb(manifest_c, "Radar Manifest", "JSON file") Rel(parser, resolver, "pass item identifier") Rel(parser, viewer, "read-only queries") Rel(parser, ring_mgr, "promote / demote commands") Rel(parser, field_editor, "set command") Rel(parser, archiver, "archive command") Rel(resolver, items_c, "load_all_items() — scan all JSON files") Rel(viewer, items_c, "read items for display") Rel(ring_mgr, items_c, "read item, write updated ring") Rel(ring_mgr, rebuilder, "trigger after write") Rel(field_editor, items_c, "read item, write updated field") Rel(field_editor, rebuilder, "trigger after write") Rel(archiver, items_c, "read item, write Hold + tag") Rel(archiver, rebuilder, "trigger after write") Rel(rebuilder, manifest_c, "writes radar.json")

L3Component — Web Dashboard

Inside index.html and dashboard.jsx (browser-side). The entire app is a single React component tree compiled in-browser by Babel from CDN — no npm, no bundler. Two view modes share the same data and filter state.

C4Component title Web Dashboard — Components (browser) Container_Boundary(dash_bound, "Web Dashboard") { Component(shell, "HTML Shell", "index.html", "Loads React 18 + ReactDOM + Babel standalone from CDN; mounts the App component into #root") Component(app, "App Component", "dashboard.jsx — App()", "Top-level component: owns filter state, fetches radar.json on mount, routes between view modes") Component(fetcher, "Data Fetcher", "dashboard.jsx — useEffect + fetch()", "Fetches /data/radar.json once on mount; falls back to hardcoded SAMPLE data if unavailable") Component(filters, "Filter Controls", "dashboard.jsx", "Ring and quadrant toggles; 'hide Discovered' and 'recent only' options; filters items for both views") Component(obs, "Observatory View", "dashboard.jsx", "Dark polar radar plot: concentric rings map to maturity levels, quadrants divide the circle; interactive hover") Component(dispatch, "Dispatch View", "dashboard.jsx", "Light editorial card grid: momentum progress bars, star counts, trend arrows, source attribution") } ContainerDb(manifest_d, "Radar Manifest", "data/radar.json") Rel(shell, app, "ReactDOM.createRoot().render(App)") Rel(app, fetcher, "useEffect on mount") Rel(fetcher, manifest_d, "fetch('/data/radar.json')") Rel(app, filters, "render; receive filter state updates") Rel(app, obs, "render when Observatory mode selected") Rel(app, dispatch, "render when Dispatch mode selected") Rel(filters, obs, "filtered item list") Rel(filters, dispatch, "filtered item list")

BehaviourSequence — Discovery Run

What happens when python runner.py is executed. The canonical-URL index lets the runner detect cross-source duplicates (same GitHub repo, different discovery source) without parsing every item file at dedup time.

sequenceDiagram autonumber actor Eng as Engineer / Cron participant R as runner.py participant S as Scraper(s) participant C as radar_core.py participant FS as data/items/ participant Raw as data/raw/ participant MF as data/radar.json Eng->>R: python runner.py R->>C: canonical_index() C->>FS: walk all JSON files, collect canonical_url fields C-->>R: {canonical_url: item_id} index loop Each registered scraper R->>S: scraper.discover() S-->>R: list of raw items R->>Raw: save_raw(scraper_name, items, today) loop Each raw item alt exists(item["id"]) — same source, already known R->>C: touch_last_seen(id, today, stars) C->>FS: update last_seen + append to stars_history else canonical_url in index — cross-source duplicate R->>C: record_also_seen(existing_id, source, url) C->>FS: append to also_seen, refresh last_seen else genuinely new technology R->>C: save_new(item) C->>FS: write data/items/source/slug.json Note over R,C: add canonical_url to in-memory index end end end R->>C: build_radar_json() C->>FS: load_all_items() — read every JSON file C->>C: compute star_trend(item) for each C->>MF: write data/radar.json R-->>Eng: done — N technologies in radar

BehaviourSequence — Curation Session

A typical weekly triage session. The engineer reviews the Discovered inbox, promotes or archives each item, and the manifest is rebuilt automatically after every change.

sequenceDiagram autonumber actor Eng as Engineer participant CLI as radar.py participant Core as radar_core.py participant FS as data/items/ participant Build as runner.build_radar_json participant MF as data/radar.json Eng->>CLI: python radar.py list --ring Discovered CLI->>Core: load_all_items() Core->>FS: walk all JSON files Core-->>CLI: all items CLI-->>Eng: grouped list printed to terminal loop For each interesting item Eng->>CLI: python radar.py show "Bun" CLI->>Core: load_all_items() + _resolve("Bun") Core-->>CLI: item JSON CLI-->>Eng: full JSON printed alt Promote to a ring Eng->>CLI: python radar.py promote "Bun" Trial CLI->>Core: _resolve("Bun") Core-->>CLI: item CLI->>FS: write item with ring="Trial" CLI->>Build: build_radar_json() Build->>FS: load_all_items() Build->>MF: write updated radar.json CLI-->>Eng: "Bun: Discovered → Trial" else Archive Eng->>CLI: python radar.py archive "StaleTech" CLI->>FS: write ring="Hold", tags+=["archived"] CLI->>Build: build_radar_json() Build->>MF: write updated radar.json CLI-->>Eng: "StaleTech: archived" else Fix metadata Eng->>CLI: python radar.py set "Bun" quadrant Tools CLI->>FS: write updated quadrant field CLI->>Build: build_radar_json() Build->>MF: write updated radar.json CLI-->>Eng: "Bun: quadrant = Tools" end end

BehaviourSequence — Build, Deploy & View

How a curated radar becomes a live dashboard — from build_site.py through static hosting to the browser rendering the interactive chart.

sequenceDiagram autonumber actor Eng as Engineer participant Build as build_site.py participant RB as runner.build_radar_json participant FS as data/items/ participant MF as data/radar.json participant Site as site/ participant Host as Static Host actor User as Team Member participant Browser as Browser participant App as React App (dashboard.jsx) Eng->>Build: python build_site.py Build->>RB: build_radar_json() RB->>FS: load_all_items() RB->>MF: write data/radar.json Build->>Site: clean + recreate site/ directory Build->>Site: copy index.html Build->>Site: copy + transform dashboard.jsx (strip export default, inject React hooks) Build->>Site: copy data/radar.json → site/data/radar.json Build-->>Eng: "site/ ready — preview with python -m http.server 8000" Eng->>Host: push site/ (GitHub Pages / Netlify drag-drop) User->>Browser: navigate to radar URL Browser->>Host: GET /index.html Host-->>Browser: index.html (loads React + Babel from CDN) Browser->>Host: GET /dashboard.jsx Host-->>Browser: dashboard.jsx source Browser->>Browser: Babel transpiles JSX in-browser Browser->>Host: fetch('/data/radar.json') Host-->>Browser: radar.json payload Browser->>App: React renders with loaded data App-->>User: interactive radar dashboard
