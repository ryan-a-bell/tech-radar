L1System Context

Who uses the radar, and what external systems does it depend on? Two personas interact with the system: an Engineer who runs and curates it (via the CLI or a local edit server in the browser), and a Team Member who views the published dashboard. Three external data sources feed the discovery pipeline, and the static dashboard is hosted on GitHub Pages.

L2Containers

The system is made of four Python processes (runner, curation CLI, site builder, edit server), a shared library, a flat-file data store, and a static React frontend. radar_core.py is the shared library every process imports. data/ is a flat filesystem store — one JSON file per technology, no database. The frontend lives in web/ and is served two ways: build_site.py bundles it into site/ for GitHub Pages (read-only), while edit_server.py serves the same files locally with editing turned on. A single runtime flag in config.js (window.RADAR_EDIT) decides which.

L3Component — Discovery Engine

Inside runner.py and the scrapers/ package. The Orchestrator drives the pipeline: run each scraper, deduplicate via a canonical-URL index built from existing items, persist new items, then assemble the dashboard manifest. Note: the RSS scraper is registered and conforms to the Scraper contract, but its discover() is still a stub that returns an empty list.

L3Component — Curation CLI

Inside radar.py and radar_core.py. The CLI is structured around a set of focused command handlers wired to an argparse subcommand parser. Every mutating command ends with a Radar Rebuild so the dashboard manifest stays in sync.

L3Component — Web Dashboard

Inside web/: index.html, config.js, and dashboard.jsx. The entire app is a single React component tree compiled in-browser by Babel from CDN — no npm, no bundler. Three view modes (Atlas is the default) share the same data, ring/quadrant/topic filters, and a detail panel. A single flag — window.RADAR_EDIT, set by config.js — gates edit mode: when on, the detail panel grows a Ring Editor that writes through to disk.

BehaviourSequence — Discovery Run

What happens when python runner.py is executed. The canonical-URL index lets the runner detect cross-source duplicates (same GitHub repo, different discovery source) without parsing every item file at dedup time.

sequenceDiagram autonumber actor Eng as Engineer / Cron participant R as runner.py participant S as Scraper(s) participant C as radar_core.py participant FS as data/items/ participant Raw as data/raw/ participant MF as data/radar.json Eng->>R: python runner.py R->>C: canonical_index() C->>FS: walk all JSON files, collect canonical_url fields C-->>R: {canonical_url: item_id} index loop Each registered scraper R->>S: scraper.discover() S-->>R: list of raw items R->>Raw: save_raw(scraper_name, items, today) loop Each raw item alt exists(item["id"]) — same source, already known R->>C: touch_last_seen(id, today, stars) C->>FS: update last_seen + append to stars_history else canonical_url in index — cross-source duplicate R->>C: record_also_seen(existing_id, source, url) C->>FS: append to also_seen, refresh last_seen else genuinely new technology R->>C: save_new(item) C->>FS: write data/items/source/slug.json Note over R,C: add canonical_url to in-memory index end end end R->>C: build_radar_json() C->>FS: load_all_items() — read every JSON file C->>C: compute star_trend(item) for each C->>MF: write data/radar.json R-->>Eng: done — N technologies in radar

BehaviourSequence — CLI Curation Session

A typical weekly triage session from the terminal. The engineer reviews the Discovered inbox, promotes or archives each item, and the manifest is rebuilt automatically after every change. (The browser path below does the same writes through edit_server.py.)

sequenceDiagram autonumber actor Eng as Engineer participant CLI as radar.py participant Core as radar_core.py participant FS as data/items/ participant Build as runner.build_radar_json participant MF as data/radar.json Eng->>CLI: python radar.py list --ring Discovered CLI->>Core: load_all_items() Core->>FS: walk all JSON files Core-->>CLI: all items CLI-->>Eng: grouped list printed to terminal loop For each interesting item Eng->>CLI: python radar.py show "Bun" CLI->>Core: load_all_items() + _resolve("Bun") Core-->>CLI: item JSON CLI-->>Eng: full JSON printed alt Promote to a ring Eng->>CLI: python radar.py promote "Bun" Trial CLI->>Core: _resolve("Bun") Core-->>CLI: item CLI->>FS: write item with ring="Trial" CLI->>Build: build_radar_json() Build->>FS: load_all_items() Build->>MF: write updated radar.json CLI-->>Eng: "Bun: Discovered → Trial" else Archive Eng->>CLI: python radar.py archive "StaleTech" CLI->>FS: write ring="Archived", stamp archived_at=today CLI->>Build: build_radar_json() Build->>MF: write updated radar.json CLI-->>Eng: "StaleTech: archived (Archived ring, 2026-06-25)" else Fix metadata Eng->>CLI: python radar.py set "Bun" quadrant Tools CLI->>FS: write updated quadrant field CLI->>Build: build_radar_json() Build->>MF: write updated radar.json CLI-->>Eng: "Bun: quadrant = Tools" end end

BehaviourSequence — Browser Curation (edit_server)

Curating in the browser instead of the terminal. edit_server.py serves the same web/ bundle as production, but intercepts /config.js to flip RADAR_EDIT on and maps /data/* to the repo-root store. Clicking a ring writes through to disk and the dashboard re-fetches — same effect as radar.py promote.

sequenceDiagram autonumber actor Eng as Engineer participant Br as Browser (dashboard.jsx) participant ES as edit_server.py participant Core as radar_core.py participant FS as data/items/ participant MF as data/radar.json Eng->>ES: python edit_server.py Eng->>Br: open http://localhost:8001/ Br->>ES: GET / , /dashboard.jsx ES-->>Br: web/ bundle (translate_path → web/) Br->>ES: GET /config.js ES-->>Br: window.RADAR_EDIT = true (intercepted) Br->>ES: GET /data/radar.json ES-->>Br: manifest (mapped to repo-root data/) Note over Br: EDIT MODE — detail panels render the Ring Editor Eng->>Br: click a ring on an item Br->>ES: POST /api/promote {id, ring} ES->>Core: id_to_path(id), load item ES->>FS: write ring (+ archived_at in/out of Archived) ES->>Core: build_radar_json() Core->>FS: load_all_items() Core->>MF: write radar.json ES-->>Br: {ok, name, ring} Br->>ES: re-fetch data/radar.json?v=N (cache-bust) ES-->>Br: updated manifest Br-->>Eng: detail panel reflects the new ring

BehaviourSequence — Build, Deploy & View

How a curated radar becomes a live dashboard. build_site.py bundles web/ into a flat site/; GitHub Actions runs this on a schedule (and on push to main) and deploys to GitHub Pages. The published config.js ships RADAR_EDIT=false, so the public dashboard is read-only by construction.

sequenceDiagram autonumber actor Eng as Engineer participant CI as GitHub Actions participant Build as build_site.py participant RB as runner.build_radar_json participant FS as data/items/ participant MF as data/radar.json participant Site as site/ participant Pages as GitHub Pages actor User as Team Member participant Browser as Browser alt Scheduled / on push to main CI->>Build: python build_site.py else Manual Eng->>Build: python build_site.py end Build->>RB: build_radar_json() RB->>FS: load_all_items() RB->>MF: write data/radar.json Build->>Site: clean + recreate site/ Build->>Site: copy web/index.html + web/config.js (RADAR_EDIT=false) Build->>Site: copy + transform web/dashboard.jsx (strip export default, inject hooks) Build->>Site: copy data/radar.json → site/data/radar.json CI->>Pages: upload + deploy site/ artifact User->>Browser: navigate to radar URL Browser->>Pages: GET /index.html Pages-->>Browser: index.html (loads React + Babel from CDN) Browser->>Pages: GET /config.js Pages-->>Browser: window.RADAR_EDIT = false (read-only) Browser->>Pages: GET /dashboard.jsx Pages-->>Browser: dashboard.jsx source Browser->>Browser: Babel transpiles JSX in-browser Browser->>Pages: fetch('data/radar.json') Pages-->>Browser: radar.json payload Browser-->>User: interactive radar dashboard (view-only)

Open work where the codebase doesn't yet reach the intended design. The structural gaps the earlier version of this page listed (the scrapers/ package, base.py, dashboard.jsx, a populated data/) are all resolved.
